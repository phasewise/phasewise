## 🚨 CRITICAL — Fix this week

### 1. Privacy & Terms pages return 307 redirect to /login
**Files:** `app/src/app/(app)/privacy/page.tsx`, `app/src/app/(app)/terms/page.tsx`, `app/src/app/(app)/layout.tsx`

**Verified:** `curl https://phasewise.io/privacy` → HTTP 307. Same for `/terms`.

**Why this is the same bug pattern as today's middleware fix:** the middleware allowlist correctly includes `/privacy` and `/terms`, but both pages are inside the `(app)/` route group whose `layout.tsx` does `redirect("/login")` for unauthenticated users. The middleware is bypassed by the layout-level guard.

**Impact:**
- GDPR/CCPA require privacy policies to be publicly accessible
- Googlebot can't index legal pages
- Footer links to /privacy and /terms (which we just deployed in landing page fix) currently send unauthenticated visitors to the login page
- Users can't read the privacy policy *before* signing up

**Fix:** Move both files out of `(app)/` into a public location:
- Create `app/src/app/(public)/layout.tsx` with shared marketing-page chrome (no auth check), OR
- Move `privacy/page.tsx` and `terms/page.tsx` directly under `app/src/app/` (no group)

The simpler fix is the second option. Both pages have minimal layout needs.

### 2. Deactivated users still have full app access
**File:** `app/src/lib/supabase/auth.ts` lines 14-17

**Verified:** `getCurrentUser()` does NOT check `isActive`. Confirmed by grepping `app/src/lib/supabase/auth.ts` — zero matches for `isActive`.

**Impact:**
- When an owner deactivates a team member via the team page, the deactivation only changes a flag — it doesn't invalidate the user's existing session
- Deactivated users can continue logging time, viewing financials, modifying data
- Their session naturally expires after Supabase's session lifetime (typically a week)
- Combined with `/api/team/members` PATCH `isActive: false` not invalidating sessions, this is a real access control gap

**Fix:** Update `getCurrentUser()` to:
```typescript
const user = await prisma.user.findUnique({
  where: { authId: supabaseUser.id },
  include: { organization: true },
});
if (!user || !user.isActive) return null;
return user;
```

The middleware already redirects null users to /login, so deactivated users will be forced out on their next request.

### 3. Form inputs not associated with labels (accessibility)
**Files:** Most form-heavy client components — confirmed by spot-check of `SubmittalsClient.tsx`, `PlantsClient.tsx`, `ClientsClient.tsx`, `projects/new/page.tsx`

**Pattern found:** `<label>...</label>` followed by `<input>` with no `id`/`htmlFor` linkage.

**Impact:**
- WCAG 2.1 Level A failure (the lowest accessibility bar)
- Screen readers can't tell users which label belongs to which input
- Keyboard users tabbing through forms hear "edit, edit, edit" instead of "Email, Password, Confirm Password"
- Voice control software ("click email field") can't find labeled inputs

**Fix template:**
```tsx
<label htmlFor="company-name">Company / Client name *</label>
<input id="company-name" value={formName} onChange={...} />
```

Roughly 12-15 forms across the app need this. ~1-2 hours of work.

## 🟠 HIGH — Fix this month

### 4. Time entries can be logged to projects the user isn't assigned to
**File:** `app/src/app/api/time/route.ts` lines 73-86

**Issue:** API verifies the phase belongs to the org but not that the user is assigned to that project. Any team member can log hours against any project in their org.

**Impact:**
- Inflated costs on projects when staff accidentally log time to wrong project
- No audit trail of "should this person have been working on this?"
- Can be abused to game profitability reports

**Fix:** Add `ProjectAssignment` lookup before allowing the time entry. If user is OWNER/ADMIN, allow regardless. Otherwise, require assignment.

### 5. Stripe webhook lacks idempotency tracking
**File:** `app/src/app/api/stripe/webhook/route.ts`

**Issue:** Stripe retries webhook deliveries on 5xx responses (and sometimes after timeouts even when we returned 200). There's no `event.id` deduplication. If `checkout.session.completed` fires twice, we send two trial-started emails.

**Impact:**
- Duplicate transactional emails to customers
- Potentially double-applied subscription state changes
- Customer support burden when customers report receiving the same email twice

**Fix:** Create a `ProcessedStripeEvent` table:
```prisma
model ProcessedStripeEvent {
  id          String   @id @default(uuid()) @db.Uuid
  stripeEventId String @unique
  processedAt DateTime @default(now())
}
```

At top of webhook handler: try to insert the event ID, on unique constraint violation return 200 immediately without processing.

### 6. Budget alert state stored in `Project.description` field (verified)
**File:** `app/src/lib/budget-alerts.ts` lines 70-87

**Issue:** Already-known: comment in code admits "This is a temporary MVP approach — proper solution would use a dedicated alerts table." The state lives in the user-editable description field as `[ALERT:WARNING_75]` markers.

**Impact:**
- User editing the project description can corrupt or delete the marker
- Concurrent description edits lose the marker
- Can't query "which projects have unsent alerts"

**Fix:** Create the `BudgetAlert` table the code comment already calls for:
```prisma
model BudgetAlert {
  id          String   @id @default(uuid()) @db.Uuid
  projectId   String   @db.Uuid
  alertLevel  String   // WARNING_75 | CRITICAL_90 | OVER_100
  sentAt      DateTime @default(now())
  project     Project  @relation(fields: [projectId], references: [id], onDelete: Cascade)
  @@unique([projectId, alertLevel])
}
```

### 7. Stripe billing page doesn't handle PAST_DUE / INCOMPLETE states
**File:** `app/src/app/(app)/settings/billing/page.tsx`

**Issue:** Page handles "success" and "canceled" query params but doesn't surface `subscriptionStatus` failure states. A user whose card fails sees a yellow "past due" badge with no remediation CTA.

**Impact:**
- Customers don't know how to fix billing failures
- Silently lost revenue (subscription cancels for failed payments without customer ever realizing)
- Particularly costly now that we're in live Stripe mode

**Fix:** Add explicit UI states for `PAST_DUE`, `INCOMPLETE`, `CANCELED`:
- PAST_DUE: prominent red banner + "Update payment method" → opens Customer Portal
- INCOMPLETE: blue banner + "Complete your subscription" → resumes Checkout
- CANCELED: amber banner + "Reactivate your subscription"

### 8. No pagination on list endpoints
**Files:**
- `app/src/app/api/compliance/route.ts`
- `app/src/app/api/invoices/route.ts`
- `app/src/app/api/submittals/route.ts`
- Likely others (plants, time-entries, etc.)

**Issue:** All return every record for the org. Fine at MVP scale, painful past ~500 records.

**Impact:**
- Eventual slow page loads as firms accumulate data
- Large network payloads on each navigation
- Vercel function timeout risk on Hobby plan past ~1k records

**Fix:** Add `?page=` and `?limit=` params with default `limit=50`. Return total count for pagination UI.

### 9. STAFF role can view all projects in the org
**Files:** `app/src/app/(app)/projects/page.tsx`, `app/src/app/(app)/projects/[id]/page.tsx`

**Issue:** Org scoping is correct, but no project-level access. STAFF users can view billing rates, fees, profit margins on projects they're not assigned to.

**Impact:**
- Privacy breach within firm (junior designer can see senior associate's billable rate via the project)
- Trust issue if firms grow past tight-knit teams

**Fix:** On project list and detail pages, filter projects to only those where:
- User is OWNER or ADMIN, OR
- User has a `ProjectAssignment` for that project

### 10. Compliance documents stored in public Supabase bucket
**File:** `app/src/app/api/compliance/upload/route.ts`

**Issue:** Files like MWELO calculations, permits, LEED docs are uploaded to a public bucket. Path is org-scoped but discoverable. Anyone who guesses the path can download the file even after the compliance item is deleted.

**Impact:**
- Information disclosure of regulatory documents
- Old deleted compliance files still accessible
- Potential client-data leak if firms upload sensitive PDFs

**Fix:**
1. Switch the bucket to private
2. Generate signed URLs (1-hour TTL) when serving file links
3. Delete files when their parent compliance record is deleted

## 🟡 MEDIUM — Improvements that compound over time

### 11. No rate limiting on any API endpoint
A motivated attacker could spam invitation creation, time entry creation, or invitation token guessing. Add Vercel Edge rate limiting or `@upstash/ratelimit`.

### 12. Invitation tokens don't have brute-force protection
`/api/invitations/[token]` GET returns 404 vs 200 distinguishably. Add a per-IP rate limit (5/min) and consider 256-bit tokens instead of UUID v4.

### 13. No CSRF protection on mutations
State-changing endpoints rely on cookie auth without CSRF token validation. Same-site cookies mitigate most attacks but explicit CSRF tokens are best practice.

### 14. JSON-LD blog Article schema hardcodes Organization as author
`app/src/app/blog/[slug]/page.tsx` line 76. Should use `post.author` if present, fall back to Organization.

### 15. Canonical URLs in blog metadata are relative
Should be absolute (`https://phasewise.io/blog/${slug}`) for clearer Google signal.

### 16. sitemap.ts and robots.ts hardcode "https://phasewise.io"
Use `process.env.NEXT_PUBLIC_APP_URL` for staging/preview compatibility.

### 17. Project number auto-increment has race condition
`app/src/app/api/projects/route.ts`. Two simultaneous project creations could skip a number. Wrap in `prisma.$transaction`.

### 18. Reset-password lacks confirmation email
Successfully changing password should email the user "your password was changed from IP X" so they can react to unauthorized changes.

### 19. Profitability report uses average billing rate
Hides the "junior staff did most work but billed at senior rate" pattern. Show per-person breakdown.

### 20. Approved timesheets aren't fully locked
`futureWeekBlocked` calculated but not enforced. Add server-side guard: if `weeklyTimesheet.status === "APPROVED"`, refuse all PATCH requests on that week's entries.

### 21. Budget alert email falls back to wrong template
`app/src/lib/budget-alerts.ts` falls back to `PAYMENT_FAILED` template if `BUDGET_ALERT` template ID isn't set. Different variable schema. Fail loudly instead.

### 22. Compliance document upload doesn't validate file type
Add allowlist (`.pdf`, `.doc`, `.docx`, `.xlsx`, `.png`, `.jpg`) and size limit (10MB). Currently anything goes including executables.

### 23. Approval/Status string fields should be enums
`Submittal.approvalStatus` (String) and `ComplianceItem.status` (String) should be Prisma enums to prevent typos.

### 24. Missing aria-labels on icon-only buttons
Many `<button><Trash2 /></button>` patterns. Screen readers say "button" — should say "Delete project".

### 25. MWELO calculator missing regulatory version stamp
Add "California Title 23, CCR, Division 2, Section 4.05" reference and link to official CalEPA calculator for validation. Currently looks authoritative but doesn't cite the regulation.

## Overall assessment

The codebase is **well-structured and ~80% production-ready**. The recurring pattern of issues is **silent failures at the boundaries** — middleware vs layout (today's bug + #1), session vs database state (#2), webhook vs business logic (#5), public vs private storage (#10). Adding explicit checks at these boundaries is the highest-leverage hardening work.

TypeScript strict mode passing + zero TODO comments + clean production build all point to a maintainable codebase. The audit didn't find any data-loss bugs or P0 security holes — the issues found are correctable hardening tasks, not red-alarm fires.

Most importantly: **the autonomous content pipeline + Stripe live mode + Search Console indexing are all working correctly.** The growth machine is sound; these audit findings are about hardening it for scale and compliance.
